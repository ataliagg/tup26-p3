# Tutorial: Del texto al árbol — Intérprete, Compilador y Patrón Visitante

## Objetivo

Este tutorial construye, paso a paso, un sistema que evalúa expresiones matemáticas como `3 + 4 * (10 - 2)`. El recorrido tiene tres etapas con un propósito pedagógico deliberado:

1. **Intérprete** — un analizador descendente recursivo (RDP) que evalúa la expresión directamente mientras la lee. Es el punto de partida más simple.
2. **Compilador** — el mismo RDP modificado para producir un árbol de sintaxis abstracta (AST). El árbol es una representación que puede procesarse múltiples veces y de múltiples formas.
3. **Visitante** — una interfaz genérica que desacopla las operaciones sobre el árbol de la estructura del árbol mismo.

A lo largo del camino, cada etapa introduce un concepto nuevo: jerarquía de clases abstractas y `record`, funciones locales recursivas, e interfaces con tipo genérico.

---

## 1. El problema: sintaxis de las expresiones

Antes de escribir código, hay que definir con precisión qué expresiones vamos a entender. Eso se hace con una **gramática formal**. La gramática no es burocracia: es exactamente el esquema que le vamos a dar al algoritmo.

### 1.1 Gramática

```
expresion  →  termino  ( ('+' | '-') termino )*
termino    →  unario   ( ('*' | '/') unario )*
unario     →  '-' unario  |  primario
primario   →  NUMERO  |  '(' expresion ')'
```

Cada regla define un **nivel de precedencia**:

| Regla       | Qué reconoce      | Precedencia |
|-------------|-------------------|-------------|
| `expresion` | `+` y `-`         | más baja    |
| `termino`   | `*` y `/`         | media       |
| `unario`    | `-` unario        | alta        |
| `primario`  | números y `(...)` | más alta    |

El hecho de que `termino` aparezca *dentro* de `expresion` —y no al revés— es lo que hace que la multiplicación tenga mayor precedencia que la suma. No es una regla especial del algoritmo: es una consecuencia directa de la estructura de la gramática.

### 1.2 Cómo la gramática sugiere el algoritmo

Cada regla de la gramática se convierte en una función. La recursión de la gramática se convierte en recursión de las funciones. El árbol de llamadas en tiempo de ejecución *es* el árbol de la expresión:

```
expresion("3 + 4 * 2")
    └── termino("3")          → 3
    └── '+' encontrado
    └── termino("4 * 2")
            └── unario("4")   → 4
            └── '*' encontrado
            └── unario("2")   → 2
            → 4 * 2 = 8
    → 3 + 8 = 11
```

Esta correspondencia directa entre gramática y código es la esencia del **análisis descendente recursivo**.

### 1.3 Decisiones de diseño

Antes de codificar cada paso, vale la pena explicitar las decisiones que tomamos:

**No hay tokenizador separado.** El analizador lee la cadena de texto directamente, carácter a carácter. En lenguajes reales, separar la tokenización del análisis simplifica el código. Aquí lo integramos para que el foco esté en el algoritmo recursivo.

**Funciones locales para las utilidades.** `SaltarEspacios`, `Detectar` y `Consumir` son funciones locales dentro del método principal. No son métodos de la clase porque no necesitan ser públicas ni reutilizadas desde afuera — son detalles de implementación del algoritmo. Las funciones locales en C# capturan variables del método que las contiene (en este caso, el texto y la posición actual) sin necesidad de pasarlos como parámetros en cada llamada.

**`record` para los nodos del AST.** Los nodos son objetos de datos inmutables: reciben sus hijos al construirse y no cambian. Eso es exactamente para lo que `record` fue diseñado — con la ventaja de que el primary constructor expone las propiedades públicamente sin declaraciones adicionales.

**Visitante sobre las clases abstractas.** En lugar de que el visitante conozca cada operador concreto (`Suma`, `Resta`, `Producto`, `Division`), los operadores le delegan dos responsabilidades a la clase abstracta: el símbolo (`Simbolo`) y la operación (`Aplicar`). El visitante queda con tres métodos genéricos en lugar de seis específicos.

## 2. Paso 1 — El intérprete: evaluación directa

El intérprete es el RDP en su forma más pura: lee la expresión de izquierda a derecha y produce un resultado numérico sin construir ninguna estructura intermedia.

### 2.1 Las funciones de apoyo

Antes de las funciones recursivas necesitamos tres utilidades que manejan el cursor sobre el texto:

```csharp
void SaltarEspacios() {
    while (pos < texto.Length && texto[pos] == ' '){
        pos++;
    }
}

// Detectar: avanza si el próximo carácter no espacial es 'c'; si no, no toca nada.
// Uso: decidir qué rama tomar — la presencia del carácter es opcional.
bool Detectar(char c) {
    SaltarEspacios();
    if (pos < texto.Length && texto[pos] == c) {
        pos++;
        return true;
    }
    return false;
}

// Consumir: igual que Detectar pero lanza excepción si el carácter no aparece.
// Uso: caracteres que deben estar, como el ')' que cierra un paréntesis.
char Consumir(char esperado) {
    SaltarEspacios();
    if (pos >= texto.Length)
        throw new FormatException( $"Se esperaba '{esperado}' pero se llegó al final de la expresión.");
    if (texto[pos] != esperado)
        throw new FormatException( $"Se esperaba '{esperado}' pero se encontró '{texto[pos]}' en posición {pos}.");
    return texto[pos++];
}

char Proximo(){
    return pos < texto.Length ? texto[pos] : '\0';
}
```

La diferencia entre `Detectar` y `Consumir` captura un matiz del algoritmo: `Detectar` *elige* una rama, `Consumir` *exige* un carácter. Esa distinción hace que los mensajes de error sean precisos y que la lógica sea explícita.

### 2.2 Las funciones recursivas

Cada regla de la gramática se convierte en una función local que retorna `double`:

```csharp
double Expresion() {
    double r = Termino();     // obligatorio: al menos un término

    while (true) {
        if(Detectar('+')) {
            r += Termino();
        } else if (Detectar('-')) {
            r -= Termino();
        } else {
            break;
        }
    }
    return r;
}

double Termino() {
    double r = Unario();

    while (true) {
        if (Detectar('*')) {
            r *= Unario();
        } else if (Detectar('/')) {
            double d = Unario();
            if (d == 0) throw new DivideByZeroException("División por cero.");
            r /= d;
        } else {
            break;
        }
    }
    return r;
}

double Unario() {
    if (Detectar('-')) return -Unario();   // recursión: '--5' → -(-(5)) = 5
    return Primario();
}

double Primario() {
    SaltarEspacios();

    if (Detectar('(')) {
        double v = Expresion();   // recursión: subexpresión entre paréntesis
        Consumir(')');
        return v;
    }

    // Leer un número: dígitos con punto decimal opcional
    int inicio = pos;
    while ((char.IsDigit(Proximo()) || Proximo() == '.')){
        pos++;
    }
    if (pos == inicio) {
        throw new FormatException($"Se esperaba un número en posición {pos}.");
    }
    return double.Parse(texto[inicio..pos]);
}
```

El bucle `while (true) / break` en `Expresion` y `Termino` implementa la **asociatividad izquierda**: `3 - 2 - 1` se evalúa como `(3 - 2) - 1 = 0`, no como `3 - (2 - 1) = 2`. La recursión de `Unario` implementa la **asociatividad derecha** del menos unario.

### 2.3 Implementación completa

```csharp
public class Interprete {
    private string texto = "";
    private int    pos   = 0;

    public double Evaluar(string expresion) {
        this.texto = expresion;
        this.pos   = 0;

        double resultado = Expresion();

        SaltarEspacios();
        if (this.pos < this.texto.Length)
            throw new FormatException( $"Carácter inesperado '{this.texto[this.pos]}' en posición {this.pos}.");

        return resultado;

        // ── Utilidades ──────────────────────────────────────────────────────

        void SaltarEspacios() {
            while (Proximo() == ' ')
                this.pos++;
        }

        bool Detectar(char c) {
            SaltarEspacios();
            if (Proximo() == c) {
                this.pos++;
                return true;
            }
            return false;
        }

        char Consumir(char esperado) {
            SaltarEspacios();
            if (Proximo() != esperado)
                throw new FormatException( $"Se esperaba '{esperado}'.");
            return this.texto[this.pos++];
        }

        char Proximo() {
            return this.pos < this.texto.Length ? this.texto[this.pos] : '\0';
        }
        // ── Gramática ───────────────────────────────────────────────────────

        double Expresion() {
            double r = Termino();
            while (true) {
                if (Detectar('+')) {
                    r += Termino();
                } else if (Detectar('-')) {
                    r -= Termino();
                } else {
                    break;
                }
            }
            return r;
        }

        double Termino() {
            double r = Unario();
            while (true) {
                if (Detectar('*')) {
                    r *= Unario();
                } else if (Detectar('/')) {
                    double d = Unario();
                    if (d == 0) throw new DivideByZeroException("División por cero.");
                    r /= d;
                } else {
                    break;
                }
            }
            return r;
        }

        double Unario() {
            if (Detectar('-')) return -Unario();
            return Primario();
        }

        double Primario() {
            SaltarEspacios();
            if (Detectar('(')) {
                double v = Expresion();
                Consumir(')');
                return v;
            }
            return Numero();
        }

        double Numero() {
            int inicio = this.pos;
            while (char.IsDigit(Proximo()) || Proximo() == '.'))
                this.pos++;
            if (this.pos == inicio)
                throw new FormatException($"Se esperaba un número en posición {this.pos}.");
            return double.Parse(this.texto[inicio..this.pos]);
    }
}
```

### 2.4 Prueba

```csharp
var interp = new Interprete();

Console.WriteLine(interp.Evaluar("3 + 4 * 2"));       // 11
Console.WriteLine(interp.Evaluar("(3 + 4) * 2"));     // 14
Console.WriteLine(interp.Evaluar("10 - 2 - 3"));      //  5  (asociatividad izquierda)
Console.WriteLine(interp.Evaluar("--5"));             //  5  (doble negación)
Console.WriteLine(interp.Evaluar("2.5 * (1 + 1.5)")); // 6.25
```

El intérprete funciona y es conciso. ¿Por qué entonces necesitaríamos algo más?

## 3. La limitación del intérprete y la utilidad del AST

El intérprete lee la expresión una sola vez y la descarta. Eso es suficiente para evaluar, pero hay cosas que no puede hacer:

- **Evaluar la misma expresión repetidas veces** (por ejemplo, `x * 2 + 1` para distintos valores de `x`).
- **Mostrar la expresión** en distintos formatos: infijo con paréntesis, árbol visual, notación polaca.
- **Optimizar** antes de ejecutar: `0 * cualquierCosa` siempre es cero.
- **Compilar** a otro lenguaje o a bytecode.

Para todo eso, necesitamos representar la expresión como una **estructura de datos** que podamos recorrer múltiples veces. Esa estructura es el **AST**.

## 4. Paso 2 — El compilador: RDP que construye un AST

### 4.1 Diseño de la jerarquía con `record`

Los nodos del AST son objetos de datos inmutables: reciben sus hijos al construirse y no cambian. `record` es la herramienta exacta para esto: el primary constructor expone las propiedades públicamente sin declaraciones adicionales.

Pero hay algo más en este diseño. Mover `Simbolo` y `Aplicar` a las clases abstractas `OperadorUnario` y `OperadorBinario` produce un beneficio directo en el visitante: ya no necesita conocer los operadores concretos. La suma, resta, producto y división pasan a ser **variantes intercambiables** de un mismo concepto.

```
Nodo                    (record abstracto)
├── Numero              (record hoja — tiene Valor)
├── OperadorUnario      (record abstracto — tiene Operando, Simbolo, Aplicar)
│       └── Negacion
└── OperadorBinario     (record abstracto — tiene Izquierdo, Derecho, Simbolo, Aplicar)
        ├── Suma
        ├── Resta
        ├── Producto
        └── Division
```

### 4.2 Implementación del AST

```csharp
// ── Nodo base ─────────────────────────────────────────────────────────────────

public abstract record Nodo {
    public abstract T Aceptar<T>(IVisitante<T> v);
}

// ── Hoja: número literal ──────────────────────────────────────────────────────

public record Numero(double Valor) : Nodo {
    public override T Aceptar<T>(IVisitante<T> v) => v.VisitarNumero(this);

    // ToString legible para depuración — sin ".0" en enteros
    public override string ToString() => Valor.ToString();
}

// ── Operador unario ───────────────────────────────────────────────────────────
// Concentra la estructura común de todos los operadores con un solo operando.
// Simbolo y Aplicar son las dos cosas que varían entre subclases.

public abstract record OperadorUnario(Nodo Operando) : Nodo {
    public abstract string Simbolo { get; }
    public abstract double Aplicar(double operando);

    // Aceptar está aquí — las subclases no lo repiten
    public override T Aceptar<T>(IVisitante<T> v) => v.VisitarUnario(this);
}

// ── Operador binario ──────────────────────────────────────────────────────────
// Concentra la estructura común de todos los operadores con dos operandos.

public abstract record OperadorBinario(Nodo Izquierdo, Nodo Derecho) : Nodo {
    public abstract string Simbolo { get; }
    public abstract double Aplicar(double izq, double der);

    public override T Aceptar<T>(IVisitante<T> v) => v.VisitarBinario(this);
}

// ── Operadores concretos ──────────────────────────────────────────────────────
// Cada uno define solo lo que lo diferencia: su símbolo y su operación.
// Aceptar, Izquierdo y Derecho vienen heredados — no se repiten.

public record Negacion(Nodo Operando) : OperadorUnario(Operando) {
    public override string Simbolo => "-";
    public override double Aplicar(double operando) => -operando;
}

public record Suma(Nodo Izquierdo, Nodo Derecho) : OperadorBinario(Izquierdo, Derecho) {
    public override string Simbolo => "+";
    public override double Aplicar(double izq, double der) => izq + der;
}

public record Resta(Nodo Izquierdo, Nodo Derecho) : OperadorBinario(Izquierdo, Derecho) {
    public override string Simbolo => "-";
    public override double Aplicar(double izq, double der) => izq - der;
}

public record Producto(Nodo Izquierdo, Nodo Derecho) : OperadorBinario(Izquierdo, Derecho) {
    public override string Simbolo => "*";
    public override double Aplicar(double izq, double der) => izq * der;
}

public record Division(Nodo Izquierdo, Nodo Derecho) : OperadorBinario(Izquierdo, Derecho) {
    public override string Simbolo => "/";
    public override double Aplicar(double izq, double der) {
        if (der == 0) throw new DivideByZeroException("División por cero.");
        return izq / der;
    }
}
```

Nótese cuánto desaparece comparado con `class`. Un operador concreto queda en tres líneas:

1. La declaración del `record` con los parámetros del constructor.
2. `Simbolo` — el símbolo textual del operador.
3. `Aplicar` — la operación matemática.

`Aceptar`, `Izquierdo`, `Derecho` y `Operando` son heredados y no se repiten en ninguna clase concreta.

### 4.3 El compilador

El compilador es el mismo RDP del intérprete, con un único cambio: donde antes retornaba `double`, ahora retorna `Nodo`. La lógica de análisis es idéntica.

```csharp
public class Compilador {
    private string texto = "";
    private int    pos   = 0;

    public Nodo Compilar(string expresion) {
        this.texto = expresion;
        this.pos   = 0;

        Nodo raiz = Expresion();

        SaltarEspacios();
        if (this.pos < this.texto.Length)
            throw new FormatException(
                $"Carácter inesperado '{this.texto[this.pos]}' en posición {this.pos}.");

        return raiz;

        // ── Utilidades (idénticas al intérprete) ────────────────────────────

        void SaltarEspacios() {
            while (Proximo() == ' ')
                this.pos++;
        }

        bool Detectar(char c) {
            SaltarEspacios();
            if (Proximo() == c) {
                this.pos++;
                return true;
            }
            return false;
        }

        char Consumir(char esperado) {
            SaltarEspacios();
            if (Proximo() != esperado)
                throw new FormatException( $"Se esperaba '{esperado}'");
            return this.texto[this.pos++];
        }

        char Proximo() {
            return this.pos < this.texto.Length ? this.texto[this.pos] : '\0';
        }
        // ── Gramática — retorna Nodo en lugar de double ──────────────────────

        Nodo Expresion() {
            Nodo izq = Termino();
            while (true) {
                if      (Detectar('+')) izq = new Suma(izq, Termino());
                else if (Detectar('-')) izq = new Resta(izq, Termino());
                else break;
            }
            return izq;
        }

        Nodo Termino() {
            Nodo izq = Unario();
            while (true) {
                if      (Detectar('*')) izq = new Producto(izq, Unario());
                else if (Detectar('/')) izq = new Division(izq, Unario());
                else break;
            }
            return izq;
        }

        Nodo Unario() {
            if (Detectar('-')) return new Negacion(Unario());
            return Primario();
        }

        Nodo Primario() {
            SaltarEspacios();
            if (Detectar('(')) {
                Nodo v = Expresion();
                Consumir(')');
                return v;
            }
            int inicio = this.pos;
            while (this.pos < this.texto.Length &&
                   (char.IsDigit(this.texto[this.pos]) || this.texto[this.pos] == '.'))
                this.pos++;
            if (this.pos == inicio)
                throw new FormatException($"Se esperaba un número en posición {this.pos}.");
            return new Numero(double.Parse( this.texto[inicio..this.pos]));
        }
    }
}
```

## 5. Paso 3 — El patrón Visitante

### 5.1 El problema

Si queremos agregar nuevas operaciones sobre el árbol — evaluar, imprimir, optimizar — y las implementamos como métodos en los nodos, cada nueva operación implica modificar todas las clases de la jerarquía.

El patrón **Visitante** invierte esa dependencia: cada operación se concentra en una clase separada, y los nodos sólo exponen un método de conexión (`Aceptar`) que no necesita cambiar cuando se agrega una nueva operación.

### 5.2 El mecanismo: doble despacho

El visitante necesita saber el **tipo concreto** de cada nodo para llamar al método correcto. La solución es el **doble despacho**:

1. El código externo llama a `nodo.Aceptar(visitante)` — el tipo real del nodo determina qué `Aceptar` se ejecuta.
2. Dentro de `Aceptar`, el nodo llama a `visitante.VisitarXxx(this)` — ahora `this` tiene el tipo concreto conocido por el compilador.

Con dos llamadas virtuales, el visitante recibe el tipo exacto sin ningún `if` ni `switch`.

### 5.3 La interfaz — 3 métodos en lugar de 6

Aquí aparece el beneficio de haber movido `Simbolo` y `Aplicar` a las clases abstractas. El visitante no necesita distinguir entre `Suma`, `Resta`, `Producto` y `Division` — todos son `OperadorBinario` y se tratan de forma uniforme:

```csharp
public interface IVisitante<T> {
    T VisitarNumero  (Numero          nodo);
    T VisitarUnario  (OperadorUnario  nodo);
    T VisitarBinario (OperadorBinario nodo);
}
```

Comparado con la versión que conoce cada operador concreto:

| Versión           | Métodos                               | Qué conoce el visitante      |
|-------------------|---------------------------------------|------------------------------|
| Por tipo concreto | 6 (`VisitarSuma`, `VisitarResta`…)    | Cada operador específico     |
| Por abstracción   | 3 (`VisitarUnario`, `VisitarBinario`) | Solo la estructura del árbol |

La segunda versión es posible porque los operadores concretos delegaron su lógica (`Aplicar`) y su nombre (`Simbolo`) a la clase abstracta. El visitante accede a esos datos a través del contrato de la clase base.

### 5.4 Los visitantes concretos

**`Evaluador`** — la lógica de evaluación queda en tres líneas. Los operadores concretos no aparecen en ninguna parte:

```csharp
public class Evaluador : IVisitante<double> {
    public double VisitarNumero(Numero n) => 
        n.Valor;

    public double VisitarUnario(OperadorUnario n) => 
        n.Aplicar(n.Operando.Aceptar(this));

    public double VisitarBinario(OperadorBinario n) =>
        n.Aplicar(n.Izquierdo.Aceptar(this), n.Derecho.Aceptar(this));
}
```

`VisitarBinario` no sabe si está evaluando una `Suma` o una `Division`. Llama a `n.Aplicar(izq, der)` y el polimorfismo de `Aplicar` hace el resto. Cuando se evalúa una `Division`, es `Division.Aplicar` quien valida la división por cero — la lógica específica está donde corresponde.

**`Impresor`** — produce la notación infija completamente parentetizada:

```csharp
public class Impresor : IVisitante<string> {
    public string VisitarNumero(Numero n) => 
        n.ToString();

    public string VisitarUnario(OperadorUnario n) =>
        $"({n.Simbolo}{n.Operando.Aceptar(this)})";
        
    public string VisitarBinario(OperadorBinario n) =>
        $"({n.Izquierdo.Aceptar(this)} {n.Simbolo} {n.Derecho.Aceptar(this)})";
}
```

El símbolo del operador llega a través de `n.Simbolo` sin que el impresor sepa el tipo concreto.

**`ImpresionArbol`** — produce el árbol indentado para visualizar la estructura:

```csharp
public class ImpresionArbol : IVisitante<string> {
    public string VisitarNumero(Numero n) {
        return n.ToString();
    }

    public string VisitarUnario(OperadorUnario n) {
        string hijo = n.Operando.Aceptar(this);
        return $"{n.Simbolo}(unario)\n└── {Indentar(hijo)}";
    }

    public string VisitarBinario(OperadorBinario n) {
        string izq = n.Izquierdo.Aceptar(this);
        string der = n.Derecho.Aceptar(this);
        return $"{n.Simbolo}\n├── {Indentar(izq, "│   ")}\n└── {Indentar(der, "    ")}";
    }

    private static string Indentar(string texto, string prefijo = "    ") {
        var lineas = texto.Split('\n');
        return lineas[0] + (lineas.Length > 1
            ? "\n" + string.Join("\n", lineas[1..].Select(l => prefijo + l))
            : "");
    }
}
```
**Optimizador** — un visitante que simplifica subexpresiones constantes:

```csharp
public class Optimizador : IVisitante<Nodo> {
    public Nodo VisitarNumero(Numero n) => n;
    public Nodo VisitarUnario(OperadorUnario n) {
        Nodo valor = n.Operando.Aceptar(this);
        if (valor is Numero num) {
            return new Numero(n.Aplicar(num.Valor));
        }
        return n with { Operando = valor };
    }
    public Nodo VisitarBinario(OperadorBinario n) {
        Nodo izq = n.Izquierdo.Aceptar(this);
        Nodo der = n.Derecho.Aceptar(this);
        if (izq is Numero numIzq && der is Numero numDer) {
            return new Numero(n.Aplicar(numIzq.Valor, numDer.Valor));
        }
        return n with { Izquierdo = izq, Derecho = der };
    }
}
### 5.5 Uso integrado

```csharp
var compilador  = new Compilador();
var evaluador   = new Evaluador();
var impresor    = new Impresor();
var arbolVisor  = new ImpresionArbol();
var optimizador = new Optimizador();

string[] expresiones = [
    "3 + 4 * 2",
    "(3 + 4) * 2",
    "10 - 2 - 3",
    "--5",
    "10 / (2 + 3)",
];

foreach (string expr in expresiones) {
    Nodo arbol = compilador.Compilar(expr);

    Console.WriteLine($"Entrada:    {expr}");
    Console.WriteLine($"Infijo:     {arbol.Aceptar(impresor)}");
    Console.WriteLine($"Resultado:  {arbol.Aceptar(evaluador)}");
    Console.WriteLine($"Optimizado: {arbol.Aceptar(optimizador).Aceptar(impresor)}");
    Console.WriteLine("Árbol:");
    Console.WriteLine(arbol.Aceptar(arbolVisor));
    Console.WriteLine();
}
```

Salida para `(3 + 4) * 2`:

```
Entrada:   (3 + 4) * 2
Infijo:    ((3 + 4) * 2)
Resultado: 14
Optimizado: (7 * 2)
Árbol:
*
├── +
│   ├── 3
│   └── 4
└── 2
```

Salida para `--5`:

```
Entrada:   --5
Infijo:    (-(-(5)))
Resultado: 5
Árbol:
-(unario)
└── -(unario)
    └── 5
```

Agregar un nuevo visitante —por ejemplo, uno que cuente la cantidad de operaciones, u otro que optimice subexpresiones constantes— sólo requiere implementar `IVisitante<T>`. Las clases del AST no se tocan.

## 6. El trade-off del diseño

Mover `Simbolo` y `Aplicar` a las clases abstractas es una elección con consecuencias:

| Situación                                           | Visitante de 3 métodos             | Visitante de 6 métodos                           |
|-----------------------------------------------------|------------------------------------|--------------------------------------------------|
| Agregar un operador nuevo (ej: `%`)                 | Solo crear el record concreto      | Crear el record y modificar todos los visitantes |
| Agregar una operación nueva (ej: optimizador)       | Solo crear un visitante            | Solo crear un visitante                          |
| Tratar `Suma` distinto de `Division` en un visitante| Requiere `is` o `switch` interno   | Directo — métodos separados                      |
| Longitud de cada visitante                          | Más corto y genérico               | Más explícito pero repetitivo                    |

La versión de 3 métodos gana cuando todas las operaciones tratan a los binarios de manera uniforme. Pierde sólo si un visitante necesita comportamiento radicalmente diferente para operadores individuales — en ese caso habría que introducir un `switch` dentro de `VisitarBinario`, lo que equivale a reconstruir parcialmente los 6 métodos.

Para este dominio —expresiones matemáticas donde cada operador es una función matemática bien definida— el diseño de 3 métodos es el más apropiado.

## 7. Qué ilustra cada mecanismo

| Mecanismo                       | Dónde aparece                                              | Efecto                                                                |
|---------------------------------|------------------------------------------------------------|-----------------------------------------------------------------------|
| `abstract record`               | `Nodo`, `OperadorUnario`, `OperadorBinario`                | Definen estructura sin poder instanciarse                             |
| `record` con primary constructor| `Numero`, `Suma`, `Negacion`, …                            | Propiedades públicas generadas automáticamente                        |
| Propiedad `abstract` en record  | `Simbolo` y `Aplicar` en las abstractas                    | Cada subclase define sólo lo que la diferencia                        |
| `override`                      | Cada clase concreta implementa `Simbolo` y `Aplicar`       | Despacho dinámico: el tipo real decide la operación                   |
| `Aceptar<T>` en la abstracción  | `OperadorUnario` y `OperadorBinario` implementan `Aceptar` | Las subclases concretas no necesitan repetirlo                        |
| Funciones locales               | `SaltarEspacios`, `Detectar`, `Consumir`, `Expresion`, …   | Capturan `this.pos` y `this.texto`; encapsulan el algoritmo           |
| Recursión de funciones locales  | `Expresion` → `Termino` → `Unario` → `Expresion`           | La pila de llamadas *es* el árbol de la expresión                     |
| `IVisitante<T>` con 3 métodos   | `Evaluador`, `Impresor`, `ImpresionArbol`                  | Nuevas operaciones sin modificar el AST                               |
| Doble despacho                  | `nodo.Aceptar(v)` → `v.VisitarBinario(this)`               | El tipo del nodo y el visitante determinan juntos el método ejecutado |
