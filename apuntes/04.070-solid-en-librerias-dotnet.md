## Introducción: ¿por qué mirar la BCL?

La **BCL** (Base Class Library) de .NET es el código que más usás en tu vida como programador C#: `List<T>`, `Dictionary<TKey, TValue>`, `Stream`, `FileStream`, `StreamReader`, `IEnumerable<T>`, LINQ, etc. Todo ese código lo diseñaron ingenieros de Microsoft que vienen aplicando principios de diseño hace décadas, y está pulido por millones de desarrolladores que lo usan todos los días.

Por eso, la BCL es un **laboratorio fantástico** para ver SOLID en la práctica. Y hay un detalle importante: **no vas a ver "SOLID puro de libro"**, porque la BCL también tiene que cargar con decisiones de compatibilidad hacia atrás, rendimiento y ergonomía. Vas a ver aplicaciones claras, pero también alguna que otra concesión pragmática. Eso es parte del aprendizaje: **los principios guían, no mandan**.

El objetivo de este tutorial es que cuando termines de leerlo, cada vez que uses `new StreamReader(new GZipStream(new FileStream(...)))` o `foreach` sobre una `List<T>`, veas el diseño detrás y entiendas por qué funciona tan bien.

---

## Las dos librerías que hacen el trabajo pesado

Para cumplir con la consigna de usar **la menor cantidad de librerías que implementen la mayor cantidad de principios**, voy a concentrarme en dos espacios de nombres:

- **`System.IO`**: todo lo relacionado con streams, archivos, lectura/escritura.
- **`System.Collections.Generic`**: listas, diccionarios, enumerables, LINQ.

Entre las dos vamos a cubrir **los cinco principios con dos ejemplos cada uno**. Ocasionalmente voy a mencionar algo de `System.Text`, `System.IO.Compression` o `Microsoft.Extensions.DependencyInjection`, pero siempre como complemento natural de las dos principales.

## S — Single Responsibility Principle

> **Una clase debe tener una sola razón para cambiar.**

La BCL es un caso de estudio magistral de SRP porque, en lugar de diseñar clases gigantes que "hagan todo con archivos", los ingenieros de .NET partieron el problema en **piezas muy finas**, cada una con una única responsabilidad.

### Ejemplo 1: el trío `FileStream` + `StreamReader` + `TextReader`

Pensemos en el problema de **leer líneas de un archivo de texto**. Suena simple, ¿no? Pero si lo descomponemos desde primeros principios, hay al menos tres responsabilidades distintas:

1. **Acceder a los bytes** que viven en un disco físico (sectores, handles del sistema operativo, buffering, permisos).
2. **Convertir bytes en caracteres** usando una codificación (UTF-8, UTF-16, Latin-1, etc.).
3. **Agrupar caracteres en líneas** (detectar `\r\n`, `\n`, `\r`).

La BCL tiene **una clase por responsabilidad**:

```csharp
// Responsabilidad 1: bytes desde/hacia un dispositivo
FileStream fs = new FileStream("datos.txt", FileMode.Open);

// Responsabilidad 2: bytes ↔ caracteres (con encoding)
StreamReader sr = new StreamReader(fs, Encoding.UTF8);

// Responsabilidad 3: caracteres → líneas (heredada de TextReader)
string? linea = sr.ReadLine();
```

Pensá qué pasaría si todo esto estuviera en una única clase `ArchivoDeTexto`:

- Si cambia el sistema de archivos (agregás S3, FTP, memoria), tenés que modificar esa clase.
- Si aparece un encoding nuevo, tenés que modificar esa misma clase.
- Si cambia la lógica de detección de líneas, la misma clase otra vez.

Tres razones distintas para cambiar → tres responsabilidades → **debe ser tres clases**. Y por eso en .NET **lo son**. Podés verlo claramente en la jerarquía: `FileStream` hereda de `Stream` (bytes), `StreamReader` hereda de `TextReader` (texto), y las dos se *componen*, no se heredan.

### Ejemplo 2: `List<T>` vs `Enumerable` (LINQ)

Mirá esta diferencia sutil pero poderosa. `List<T>` tiene la responsabilidad de **almacenar** elementos en memoria (agregar, quitar, indexar, capacidad, redimensionamiento interno). Nada más.

Pero uno quiere hacer cosas sobre las colecciones: filtrar, mapear, ordenar, agrupar. ¿Esas operaciones son responsabilidad de `List<T>`? **No.** Si lo fueran, cada colección (`List<T>`, `Queue<T>`, `Stack<T>`, `HashSet<T>`, etc.) tendría que reimplementar Where, Select, OrderBy... y cambiar la lógica de filtrado obligaría a tocar **todas** las colecciones.

Los diseñadores de LINQ pusieron esas operaciones en una clase **estática separada**: `System.Linq.Enumerable`. Todos los métodos son **métodos de extensión** sobre `IEnumerable<T>`:

```csharp
// Enumerable.cs, simplificado
public static class Enumerable
{
    public static IEnumerable<TSource> Where<TSource>(
        this IEnumerable<TSource> source,
        Func<TSource, bool> predicate) { /* ... */ }

    public static IEnumerable<TResult> Select<TSource, TResult>(
        this IEnumerable<TSource> source,
        Func<TSource, TResult> selector) { /* ... */ }
}
```

Resultado:

- `List<T>` solo almacena. Cambia si cambia cómo se almacena.
- `Enumerable` solo opera. Cambia si cambia cómo se opera.
- Un dev que escriba su propia colección y haga `: IEnumerable<T>` hereda **gratis** todo LINQ.

Es SRP llevado al extremo más elegante: **una clase no implementa operaciones que no le son propias**.

## O — Open/Closed Principle

> **Abierto a la extensión, cerrado a la modificación.**

Este principio es el que más brilla en .NET, porque **toda la arquitectura de streams** está construida sobre él.

### Ejemplo 1: `Stream` como jerarquía extensible

La clase base `System.IO.Stream` es una abstracción mínima: "un lugar de donde puedo leer bytes y a donde puedo escribir bytes". Su contrato es básicamente:

```csharp
public abstract class Stream : IDisposable
{
    public abstract bool CanRead { get; }
    public abstract bool CanWrite { get; }
    public abstract bool CanSeek { get; }
    public abstract int Read(byte[] buffer, int offset, int count);
    public abstract void Write(byte[] buffer, int offset, int count);
    public abstract long Position { get; set; }
    public abstract long Length { get; }
    // ...
}
```

A partir de esa base, la BCL (y cualquier librería de terceros) agrega implementaciones **sin modificar `Stream`**:

- `FileStream` — bytes desde/hacia un archivo.
- `MemoryStream` — bytes en un `byte[]` en memoria.
- `NetworkStream` — bytes desde/hacia un socket.
- `GZipStream`, `DeflateStream`, `BrotliStream` — compresión/descompresión *envolviendo* otro stream.
- `CryptoStream` — cifrado/descifrado envolviendo otro stream.
- `BufferedStream` — agrega buffering a cualquier otro stream.

El detalle clave: `GZipStream` **no es un archivo comprimido**. Es un `Stream` que, cuando le pedís bytes, los saca de **otro stream** (el que vos le pasás) y los descomprime al vuelo. Esto permite componer:

```csharp
using var fs        = new FileStream("datos.gz", FileMode.Open);
using var descomp   = new GZipStream(fs, CompressionMode.Decompress);
using var lector    = new StreamReader(descomp, Encoding.UTF8);

string? linea;
while ((linea = lector.ReadLine()) != null)
    Console.WriteLine(linea);
```

Cada línea agrega una capa de comportamiento **sin tocar las clases de abajo**. Si mañana necesitás leer desde un archivo comprimido con GZip y además cifrado con AES, el patrón es idéntico: metés un `CryptoStream` en el medio. Ninguna de las clases existentes sabe de la existencia de las otras, pero todas **colaboran** porque dependen de la misma abstracción: `Stream`.

Esto es el **patrón Decorator** en su forma más limpia, y es OCP aplicado: se pueden agregar capacidades nuevas (cifrado, compresión, buffering, throttling) **sin modificar una sola línea** de `Stream`, `FileStream` ni `StreamReader`.

### Ejemplo 2: LINQ sobre `IEnumerable<T>`

El mismo patrón en otro dominio. LINQ tiene cerca de 50 operadores (`Where`, `Select`, `OrderBy`, `GroupBy`, `Join`, ...). Todos trabajan sobre `IEnumerable<T>` y devuelven `IEnumerable<T>`.

¿Qué te permite esto? **Agregar operadores nuevos sin tocar la BCL**. Podés escribir tu propio método de extensión:

```csharp
public static class MisExtensiones
{
    public static IEnumerable<T> SaltearCada<T>(this IEnumerable<T> fuente, int n)
    {
        int i = 0;
        foreach (var x in fuente)
            if (i++ % n != 0) yield return x;
    }
}

// Uso:
var pares = numeros.Where(x => x % 2 == 0).SaltearCada(3).Select(x => x * 10);
```

`SaltearCada` se compone naturalmente con `Where` y `Select` porque todos hablan el mismo idioma: `IEnumerable<T>`. Microsoft no previó tu operador custom, y sin embargo funciona. Eso es OCP: **el sistema se extiende sin modificarse**.

## L — Liskov Substitution Principle

> **Una subclase debe poder reemplazar a su clase base sin romper el código que usa la clase base.**

LSP es más sutil y, justamente, acá es donde vemos las **concesiones pragmáticas** de la BCL. Hay ejemplos muy limpios, y hay uno muy famoso de violación que vale la pena entender.

### Ejemplo 1: `IEnumerable<T>` y `foreach` (cumple LSP)

`IEnumerable<T>` es la abstracción más usada de .NET. Su contrato es simple: "podés pedirme un enumerador, y el enumerador te devuelve elementos de a uno".

```csharp
public interface IEnumerable<out T> : IEnumerable
{
    IEnumerator<T> GetEnumerator();
}
```

Un `foreach` no sabe si está iterando un `List<T>`, un `HashSet<T>`, un `Dictionary<K,V>.Values`, un archivo leído línea por línea con `File.ReadLines()`, o el resultado de un `Where().Select()`. **Y no le importa.** Cualquier tipo que implemente `IEnumerable<T>` cumple el contrato:

```csharp
void Imprimir<T>(IEnumerable<T> fuente)
{
    foreach (var item in fuente)
        Console.WriteLine(item);
}

Imprimir(new List<int> { 1, 2, 3 });           // funciona
Imprimir(new HashSet<int> { 1, 2, 3 });        // funciona
Imprimir(File.ReadLines("datos.txt"));         // funciona (¡streaming!)
Imprimir(Enumerable.Range(1, 1_000_000));      // funciona (lazy)
```

**Todas las implementaciones de `IEnumerable<T>` son intercambiables** desde el punto de vista del cliente. Esto es LSP en su forma más pura y es la razón por la que LINQ, `foreach`, `async foreach` y toda la maquinaria funcional de C# simplemente *funcionan* con cualquier fuente de datos.

### Ejemplo 2: `Stream` — el caso interesante (violación parcial)

Acá viene la parte honesta. `Stream` define métodos como `Seek`, `Position` y `Length`. Pero no todos los streams pueden posicionarse: un `NetworkStream` o un `GZipStream` son **secuenciales por naturaleza**, no podés "volver atrás" en una conexión TCP.

¿Qué hicieron en la BCL? Dos cosas:

**Primero, propiedades de capacidad**. Cada stream declara explícitamente qué puede hacer:

```csharp
public abstract bool CanRead { get; }
public abstract bool CanWrite { get; }
public abstract bool CanSeek { get; }
```

**Segundo, excepciones documentadas**. Si llamás a `Seek` en un stream que no soporta seek, tira `NotSupportedException`.

```csharp
using var red = new NetworkStream(socket);
red.Position = 0; // 💥 NotSupportedException
```

Estrictamente hablando, **esto viola LSP**. Un método que recibe un `Stream` no puede confiar en que `Seek` funcione; tiene que chequear `CanSeek` primero. La "sustituibilidad" del contrato base no es completa.

¿Por qué lo hicieron así? Porque la alternativa era partir `Stream` en `ISeekableStream` vs `IForwardOnlyStream`, y eso habría complicado enormemente el uso común. Los ingenieros de .NET priorizaron la **ergonomía** sobre la pureza del principio. Es una concesión consciente.

**La lección que te tenés que llevar**: cuando veas este patrón (capacidades declaradas + excepciones para lo no soportado) en código ajeno o propio, sabé que es un compromiso. A veces es razonable, a veces es una señal de que tu jerarquía está mal pensada. Si tu subtipo tira `NotSupportedException` en la mitad de los métodos, probablemente tendrías que dividir la interfaz. Lo cual nos lleva al siguiente principio.

## I — Interface Segregation Principle

> **Los clientes no deben depender de interfaces que no usan.**

.NET segregó sus colecciones en interfaces chicas y específicas. Vale la pena mirar cómo.

### Ejemplo 1: la jerarquía de `System.Collections.Generic`

Mirá cómo están separadas las capacidades de las colecciones:

```csharp
public interface IEnumerable<out T>              // solo iterar
{
    IEnumerator<T> GetEnumerator();
}

public interface IReadOnlyCollection<out T>      // + contar
    : IEnumerable<T>
{
    int Count { get; }
}

public interface IReadOnlyList<out T>            // + acceder por índice (readonly)
    : IReadOnlyCollection<T>
{
    T this[int index] { get; }
}

public interface ICollection<T>                  // + modificar
    : IEnumerable<T>
{
    int Count { get; }
    void Add(T item);
    bool Remove(T item);
    void Clear();
    bool Contains(T item);
    bool IsReadOnly { get; }
}

public interface IList<T>                        // + indexar y modificar
    : ICollection<T>
{
    T this[int index] { get; set; }
    int IndexOf(T item);
    void Insert(int index, T item);
    void RemoveAt(int index);
}
```

Esto te permite expresar **exactamente** lo que necesita cada método:

```csharp
// Solo necesito iterar
void Imprimir<T>(IEnumerable<T> items) { /* ... */ }

// Necesito saber cuántos hay, pero no modificar
string Describir<T>(IReadOnlyCollection<T> items)
    => $"Tengo {items.Count} elementos";

// Necesito leer por índice sin modificar (ideal para API pública)
T ObtenerTercero<T>(IReadOnlyList<T> items) => items[2];

// Necesito poder agregar
void AgregarVarios<T>(ICollection<T> items, IEnumerable<T> nuevos)
{
    foreach (var n in nuevos) items.Add(n);
}
```

Pensá qué pasaría si .NET solo tuviera una `ICollection<T>` gigante con todo adentro. Cada método que solo quiere iterar estaría aceptando un parámetro con más poder del que necesita: podría modificarlo, vaciarlo, etc. El contrato del método sería **mentiroso**. Con interfaces segregadas, el tipo del parámetro **documenta qué vas a hacer con él**.

Caso concreto: si tenés una API pública, aceptar `IReadOnlyList<Pedido>` en lugar de `List<Pedido>` es una forma de **comunicar y garantizar** que el método no va a modificar la lista del llamador. Es contrato y documentación al mismo tiempo.

### Ejemplo 2: `IDisposable` vs `IAsyncDisposable`

Acá ISP se ve en chiquito pero con mucha claridad. El contrato "liberar recursos" empezó siendo:

```csharp
public interface IDisposable
{
    void Dispose();
}
```

Pero con la llegada de async/await, algunos recursos necesitan liberarse con operaciones asincrónicas (por ejemplo, un `FileStream` que tiene que hacer flush al disco, o una conexión de red que tiene que mandar un paquete de cierre). ¿Qué hicieron? **No modificaron `IDisposable`**. Crearon una interfaz nueva:

```csharp
public interface IAsyncDisposable
{
    ValueTask DisposeAsync();
}
```

Una clase puede implementar una, la otra, o las dos:

```csharp
public class FileStream : Stream, IDisposable, IAsyncDisposable
{
    public void Dispose() { /* sincrónico */ }
    public ValueTask DisposeAsync() { /* asincrónico */ }
}
```

Y `using` en C# ahora soporta las dos:

```csharp
using var fs = new FileStream("a.txt", FileMode.Open);          // usa Dispose()
await using var fs2 = new FileStream("a.txt", FileMode.Open);   // usa DisposeAsync()
```

Si `IDisposable` hubiera tenido siempre los dos métodos, **toda clase dispose-able** habría tenido que implementar un `DisposeAsync` que nadie pidió. Con ISP, cada clase declara **solo lo que ofrece**.

## D — Dependency Inversion Principle

> **Depender de abstracciones, no de implementaciones concretas.**

### Ejemplo 1: `StreamReader` depende de `Stream`, no de `FileStream`

Mirá la firma del constructor:

```csharp
public StreamReader(Stream stream, Encoding encoding) { /* ... */ }
```

El parámetro es `Stream` — la abstracción. No `FileStream`, no `NetworkStream`. Esto es DIP puro y es **la razón por la que podés hacer todo esto**:

```csharp
// Leer desde un archivo
var reader1 = new StreamReader(new FileStream("a.txt", FileMode.Open));

// Leer desde un archivo comprimido
var reader2 = new StreamReader(
    new GZipStream(new FileStream("a.gz", FileMode.Open),
                   CompressionMode.Decompress));

// Leer desde una respuesta HTTP
using var http = new HttpClient();
var reader3 = new StreamReader(await http.GetStreamAsync("https://..."));

// Leer desde memoria (típico en tests)
var bytes = Encoding.UTF8.GetBytes("hola\nmundo");
var reader4 = new StreamReader(new MemoryStream(bytes));

// Leer desde un string en memoria, sin Stream (StringReader hereda TextReader)
var reader5 = new StringReader("hola\nmundo");
```

`StreamReader` **no sabe ni le importa** de dónde vienen los bytes. Depende de la abstracción `Stream`, y eso le da una flexibilidad enorme. Y fijate el regalo que trae para los tests: podés probar cualquier lógica que lea texto pasándole un `MemoryStream` con datos fabricados, sin tocar el disco. Eso no es casualidad — es consecuencia directa de haber invertido la dependencia.

### Ejemplo 2: `IServiceCollection` y el contenedor de DI de .NET

En ASP.NET Core, DIP no es una opción sino **la forma esperada** de trabajar. Mirá un controller típico:

```csharp
public class TarjetasController : ControllerBase
{
    private readonly ILogger<TarjetasController> _logger;
    private readonly IRepositorioTarjetas _repo;
    private readonly INotificador _notificador;

    public TarjetasController(
        ILogger<TarjetasController> logger,
        IRepositorioTarjetas repo,
        INotificador notificador)
    {
        _logger = logger;
        _repo = repo;
        _notificador = notificador;
    }

    [HttpPost("debitar")]
    public async Task<IActionResult> Debitar(string numero, decimal monto)
    {
        var tarjeta = await _repo.ObtenerAsync(numero);
        tarjeta.Debitar(monto);
        await _repo.GuardarAsync(tarjeta);
        await _notificador.NotificarAsync(tarjeta.Email, "Débito", $"${monto}");
        _logger.LogInformation("Débito de {Monto} en {Numero}", monto, numero);
        return Ok();
    }
}
```

El controller **no hace `new` de nada**. Solo depende de tres interfaces: `ILogger<T>`, `IRepositorioTarjetas`, `INotificador`. Cuál implementación concreta se usa se decide en un único lugar, `Program.cs`:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddScoped<IRepositorioTarjetas, RepositorioSqlServer>();
builder.Services.AddScoped<INotificador, NotificadorEmailSmtp>();
builder.Logging.AddConsole();
```

Si mañana cambiás SQL Server por PostgreSQL, o email por SMS, o logs por un servicio remoto, **cambiás una línea**. El controller no se entera.

Y como corolario que ata todo el tutorial: `ILogger<T>` es en sí mismo un ejemplo precioso de todos los principios juntos. Es una interfaz chica (**ISP**) que permite implementaciones intercambiables (**LSP**, **DIP**), donde podés agregar loggers nuevos (archivos, console, Seq, Datadog, lo que quieras) sin modificar nada existente (**OCP**), y cada implementación tiene una única responsabilidad: registrar logs (**SRP**).

## Mini caso integrador: leer un archivo comprimido con LINQ

Para cerrar, armemos un pipeline que procesa un archivo JSON-Lines comprimido con GZip, filtra registros y los agrega. Fijate cómo, **sin pensar explícitamente en SOLID**, estás usando los cinco principios porque la BCL te los ofrece de fábrica:

```csharp
using System.IO;
using System.IO.Compression;
using System.Text;
using System.Text.Json;

public record Movimiento(string Tarjeta, string Tipo, decimal Monto);

public static class Procesador
{
    public static decimal TotalDebitosGold(string rutaArchivoGz)
    {
        // [DIP] StreamReader depende de Stream, no de archivos
        // [OCP] Apilamos GZipStream sobre FileStream sin tocar nada
        // [SRP] Cada capa tiene una única responsabilidad
        using var archivo   = new FileStream(rutaArchivoGz, FileMode.Open);
        using var descomp   = new GZipStream(archivo, CompressionMode.Decompress);
        using var lector    = new StreamReader(descomp, Encoding.UTF8);

        // [LSP] El enumerable de abajo es intercambiable con cualquier IEnumerable<string>
        // [ISP] Solo necesitamos iterar (IEnumerable), no modificar
        return LeerLineas(lector)
            .Select(linea => JsonSerializer.Deserialize<Movimiento>(linea)!)
            .Where(m => m.Tipo == "debito" && m.Tarjeta.StartsWith("GOLD"))
            .Sum(m => m.Monto);
    }

    private static IEnumerable<string> LeerLineas(TextReader lector)
    {
        string? linea;
        while ((linea = lector.ReadLine()) != null)
            yield return linea;
    }
}
```

Ocho líneas de lógica real. Cada decisión de diseño de la BCL te está ayudando:

- Si mañana el archivo viene de una API en vez de disco, cambiás `FileStream` por el stream de `HttpClient`. **El resto no se toca.**
- Si el archivo usa Brotli en vez de GZip, cambiás `GZipStream` por `BrotliStream`. **El resto no se toca.**
- Si en los tests querés probar con un string en memoria, usás `new StringReader(texto)` directamente. **El resto no se toca.**
- Si aparece una regla nueva (agrupar por fecha, por ejemplo), agregás un `.GroupBy()` al pipeline. **Nada existente se toca.**

Esa sensación de "todo compone como Lego" no es un accidente. Es el resultado de haber aplicado SOLID consistentemente durante 20 años de evolución de la plataforma.

## Resumen

| Principio | Dónde verlo en .NET | Qué aprender de ahí |
|-----------|---------------------|---------------------|
| **SRP** | `FileStream` / `StreamReader` / `TextReader` separados; `List<T>` vs `Enumerable` | Una clase, una razón para cambiar. Componé, no acumules. |
| **OCP** | `Stream` y sus decoradores (`GZipStream`, `CryptoStream`, `BufferedStream`); operadores LINQ | Diseñá abstracciones mínimas y dejá que otros extiendan sin modificar. |
| **LSP** | `IEnumerable<T>` ✔ sustitución perfecta; `Stream` con `CanSeek` es una concesión pragmática | Si tu subtipo tira `NotSupportedException`, pensá si no deberías segregar la interfaz. |
| **ISP** | `IEnumerable` → `IReadOnlyCollection` → `IReadOnlyList` → `ICollection` → `IList`; `IDisposable` vs `IAsyncDisposable` | Muchas interfaces chicas > una grande. El tipo del parámetro documenta la intención. |
| **DIP** | `StreamReader(Stream)`; el contenedor de DI con `IServiceCollection` | Depender de `Stream`, no de `FileStream`. Depender de `ILogger<T>`, no de un archivo concreto. |

### Conclusiones para llevarte

1. **La BCL es un libro de diseño abierto.** Cada vez que dudes cómo estructurar algo, mirá cómo resolvieron un problema parecido en `System.IO` o `System.Collections.Generic`. El código fuente está en [github.com/dotnet/runtime](https://github.com/dotnet/runtime) y vale muchísimo leerlo.
2. **Los principios se potencian entre sí.** Un buen diseño no aplica uno a la vez; aplica todos al mismo tiempo y se nota en cómo el código "compone". El ejemplo de leer un `.json.gz` en 8 líneas muestra eso.
3. **Pragmatismo sobre pureza.** La BCL hace concesiones cuando cumplir estrictamente un principio complicaría el uso cotidiano (ej: `CanSeek` en `Stream`). Es un recordatorio de que SOLID guía, no impone.
4. **Lo que escribís hoy tiene que componer con lo que van a escribir otros mañana.** Si tus abstracciones son finas y cohesivas, tus compañeros (o vos en tres meses) van a poder extenderlas sin pedirte permiso.
